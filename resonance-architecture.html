Chapter 1

The Architecture in One Breath

Planetary Resonance is a distributed matching network. You describe what you need or what you offer — a skill, a room, a service, a connection — and the network finds the person on the other side. Unlike every centralized platform that exists today, your raw data never leaves your device. The network only sees mathematical shadows — vector embeddings — that are close enough to find matches but too lossy to reconstruct what you actually said.

The architecture has three tiers, each with a distinct responsibility and a distinct trust boundary:

Design Principle

Relays are functions, not servers. Any node can become a relay temporarily. No one owns a relay. Relays rotate, and the vector index is sharded across multiple relays so no single relay ever holds the complete picture. This prevents the "instance admin" problem that plagues federated systems like Mastodon.

The key insight: the relay introduces you, then gets out of the way. Like a matchmaker who makes introductions but isn't present for the conversation. The relay sees enough to know you might be a good match (noisy embeddings), but never enough to know what you actually need (raw data), and once the match is made, the relay is no longer involved.

Chapter 3

Personal Node: Your Sovereign Core

The personal node is the atomic unit of Resonance. It runs on your device — a phone, laptop, a Raspberry Pi in your closet, a small VPS you control. It is the only place where your raw data exists. Everything else in the network works with mathematical approximations of that data.

Internal Architecture

Encrypted local database (SQLite + libsodium). Holds your raw needs, offers, profile data, and match history. Never synced. Never uploaded. Encrypted at rest with your key.

Local inference of open embedding models (BGE-small, E5, Nomic-embed, GTE). Converts your text inputs into 384-768 dimensional vectors. Runs on CPU — no GPU required. ~50ms per embedding.

Embedding Model Selection

The personal node ships with a small, efficient embedding model that runs on consumer hardware. The choice of model is not locked — nodes can use different models as long as they share a common vector space or a translation layer is available.

Model Dimensions Size CPU Latency Quality (MTEB) Best For

Nomic-embed-text 768 137 MB ~50ms 66.5 Default choice, good balance
E5-base-v2 768 109 MB ~45ms 64.8 Multilingual needs
GTE-large 1024 326 MB ~120ms 68.4 High-fidelity nodes with resources
Implementation Note

Nodes using different embedding models need a projection layer — a small learned linear map that aligns vector spaces. This is trained on public benchmark data and shipped with the node software. Relay shards maintain separate HNSW indexes per vector space family, or use projected vectors in a unified space.

Chapter 4

Relay Mesh: Discovery Without Dominion

The relay mesh is the nervous system of Resonance. It enables network-wide discovery — finding your match not just among your neighbors but across the entire network — without any single entity controlling the index. Relays are ephemeral, rotating, and sharded.

What a Relay Does

A relay node performs exactly three functions: it receives perturbed embeddings from personal nodes, it maintains a shard of the network's approximate nearest-neighbor (ANN) index, and it sends match notifications when two embeddings in its shard are close enough in vector space. That's it. It does not store raw data. It does not authenticate users. It does not moderate content. It does not decide who sees what. It is a pure matching function.

Relay Rotation

No node is a permanent relay. Relay duty is assigned through a verifiable random function (VRF) seeded by the network's distributed randomness beacon. Every epoch (configurable, default 24 hours), the relay set rotates:

// Relay selection via VRF
function selectRelays(epoch, nodePool, targetCount) {
  const seed = deriveEpochSeed(epoch, networkRandomness)
  const eligible = nodePool.filter(n => n.uptime > 0.95 && n.stake >= MIN_STAKE)
  const selected = vrfSelect(eligible, seed, targetCount)

  // Handoff: old relays transfer index shards to new relays
  for (const relay of selected) {
    const shard = assignShard(relay, seed, SHARD_COUNT)
    relay.acceptShard(shard, previousEpochRelays[shard])
  }
  return selected
}
Why Rotation Matters

If relays were permanent, they'd accumulate power. A relay that runs for months would build a complete picture of who's looking for what, could analyze matching patterns, could selectively censor matches. Rotation prevents this: by the time a relay could learn anything useful, it's no longer a relay. The knowledge evaporates.

Index Sharding

The network's ANN index is never held by a single relay. It is split into S shards (default: 16 at launch, scaling with network size), where each shard covers a region of the vector space. When a personal node publishes a perturbed embedding, a deterministic function maps it to the correct shard(s):

// Shard assignment via locality-sensitive hashing
function assignToShards(embedding, shardCount, replicationFactor = 3) {
  // LSH projects the embedding into shard buckets
  const primaryShard = lshHash(embedding) % shardCount

  // Replicate to adjacent shards for boundary matches
  const shards = [primaryShard]
  for (let i = 1; i < replicationFactor; i++) {
    shards.push((primaryShard + i) % shardCount)
  }
  return shards // embedding goes to 3 shards
}

Each shard is maintained by R relay nodes (default: 3) for redundancy. This means each embedding is stored on 3×3 = 9 relay instances across the network. No single relay holds more than 1/S of the total index, and each shard has enough replicas to survive relay failures.

HNSW Index Per Shard

Within each shard, the relay builds a Hierarchical Navigable Small World (HNSW) graph — the state-of-the-art structure for approximate nearest-neighbor search. HNSW provides sub-millisecond search over millions of vectors with ~95% recall.

Parameter Value Rationale
M (connections per layer) 16 Balance between recall and memory. Higher M = better recall, more RAM
ef_construction 200 Index build quality. Higher = better graph but slower insertion
ef_search 100 Query quality. Tuned per latency target (50 for real-time, 200 for async)
Distance metric Cosine similarity Standard for text embeddings. Normalized vectors, angle-based comparison
Max vectors per shard ~2M Keeps memory under 8GB per shard relay. Scale by adding shards

Relay Economics

Running a relay costs real resources — RAM for the HNSW index, bandwidth for receiving embeddings and sending notifications, compute for index updates. Following the compensation model from our analysis, relay operators receive:

Chapter 5

Data Flow: From Need to Resonance

This is the critical path — the journey from "I need something" to "someone has it." Every step is designed so that privacy increases as data moves further from the user, and fidelity increases as the match gets closer to confirmation.

The Complete Flow

Step 1: Create & Publish

The user describes their need or offer in natural language. The personal node converts it into a vector embedding, applies differential privacy noise, signs it, and sends it to the appropriate relay shards.

// Publishing flow on the personal node
async function publishNeed(rawText, privacyLevel) {
  // 1. Raw text never leaves this function
  const embedding = await model.encode(rawText) // local inference

  // 2. Apply differential privacy
  const epsilon = privacyToEpsilon(privacyLevel) // low=0.1, med=1.0, high=5.0
  const noisy = addLaplaceNoise(embedding, epsilon)
  const perturbed = normalize(noisy) // re-normalize to unit sphere

  // 3. Sign with DID key
  const signed = signEmbedding(perturbed, myDID, myPrivKey)

  // 4. Route to correct relay shards
  const shards = assignToShards(perturbed, SHARD_COUNT)
  for (const shardId of shards) {
    const relays = getRelaysForShard(shardId)
    await publishToRelays(relays, signed, shardId)
  }

  // 5. Store locally for later confirmation
  localDB.save({ rawText, embedding, perturbed, shards, timestamp: now() })
}
Privacy Budget

Each publication consumes from a privacy budget. Publishing the same need repeatedly with different noise would allow an observer to average out the noise and recover the true embedding. The personal node tracks cumulative ε spend and warns the user when re-publishing would degrade privacy below their configured threshold.

Step 2a: Async Matching (Bulletin Board Mode)

Your perturbed embedding sits in the relay's HNSW index. When a new embedding arrives that's close enough in vector space, the relay sends both parties a match notification.

// Relay-side matching loop
async function onNewEmbedding(incoming) {
  // Search existing index for neighbors
  const neighbors = hnswIndex.search(incoming.vector, {
    k: 10, // top 10 candidates
    ef: 100, // search quality parameter
    threshold: 0.72 // minimum cosine similarity
  })

  // Notify matched parties
  for (const match of neighbors) {
    const notification = {
      matchId: generateMatchId(),
      similarity: match.score,
      partnerDID: match.did, // pseudonymous, not real identity
      relayShard: this.shardId,
      expiry: now() + MATCH_TTL
    }
    await notifyNode(incoming.did, notification)
    await notifyNode(match.did, notification)
  }

  // Insert into index for future matches
  hnswIndex.insert(incoming.vector, incoming.did)
}

The relay never knows what the match is about. It only knows that two vectors are close in a high-dimensional space. The semantic meaning — whether it's a housing match, a skill match, or a healthcare match — is invisible to the relay.

Step 2b: Live Search (Real-Time Mode)

For urgent needs, the personal node can perform a live query across the relay mesh. The query is a perturbed embedding sent as a one-shot search — it is not stored in the index.

// Live search from personal node
async function liveSearch(rawText, privacyLevel, maxResults = 5) {
  const embedding = await model.encode(rawText)
  const perturbed = addLaplaceNoise(embedding, privacyToEpsilon(privacyLevel))

  // Query all shards in parallel (or target specific shards)
  const targetShards = assignToShards(perturbed, SHARD_COUNT)
  const results = await Promise.all(
    targetShards.map(shardId => {
      const relays = getRelaysForShard(shardId)
      return queryRelay(relays[0], perturbed, { k: maxResults, ephemeral: true })
    })
  )

  // Merge, deduplicate, rank by similarity
  return mergeAndRank(results.flat(), maxResults)
}

The ephemeral: true flag tells the relay not to insert this embedding into its index. It's a read-only query. The relay processes it, returns results, and forgets it. This is important for privacy: a live search doesn't leave a trace in the network.

Step 3: Match Confirmation

When both parties receive a match notification and choose to explore it, they enter the confirmation protocol — a direct P2P exchange of higher-fidelity information to verify the match is real.

3.1
Mutual Consent
Both nodes signal interest via the relay
→
3.2
Key Exchange
X25519 Diffie-Hellman via relay (relay can't read)
→
3.3
Direct Probe
Exchange full-fidelity embeddings, compute true similarity
→
3.4
Confirm / Reject
If similarity > threshold, proceed to direct channel

This two-phase approach (noisy match → precise confirmation) means false positives from noise are caught before any real information is exchanged. The perturbation that protects privacy at the relay level is resolved at the personal level only between consenting parties.

Chapter 6

Direct Channel: Where Value Happens

The direct channel is where Resonance stops being infrastructure and starts being useful. Once two nodes confirm a match, they establish an encrypted peer-to-peer connection that bypasses the relay mesh entirely. The relay facilitated the introduction; now it steps aside.

Channel Properties

Encryption
🔐

End-to-End Encrypted

X25519 key exchange, ChaCha20-Poly1305 symmetric cipher. Forward secrecy via ephemeral keys per session. No relay, no server, no third party can read the channel.

Consent
✋

Progressive Disclosure

You control what to share and when. Start with just the category ("I offer plumbing services"). Progress to details ("available weekdays in Barcelona, €40/h"). Share contact info only when ready.

Protocol
🔄

Structured Exchange

The channel protocol supports typed messages: offers, counteroffers, confirmations, reviews. Not free-form chat (that comes later if both want it). Structured to minimize friction.

Trust
⭐

Mutual Reputation

After an exchange, both parties can leave a signed, encrypted review. Reviews are stored locally and shared selectively — you choose who sees your reputation. No global public rating system.

Connection Protocol

// Direct channel establishment
async function openDirectChannel(matchNotification) {
  // 1. Signal consent via relay (encrypted, relay can't read)
  const consent = encryptForPeer({
    matchId: matchNotification.matchId,
    accept: true,
    ephemeralPubKey: generateX25519Keypair().publicKey
  }, matchNotification.partnerDID)
  await relay.forward(consent, matchNotification.partnerDID)

  // 2. Wait for partner's consent + ephemeral key
  const partnerConsent = await waitForConsent(matchNotification.matchId)

  // 3. Derive shared secret (Diffie-Hellman)
  const sharedSecret = x25519(myEphemeralPrivKey, partnerConsent.ephemeralPubKey)

  // 4. Establish direct P2P connection (QUIC/WebRTC)
  const channel = await connectPeer(partnerConsent.endpoints, {
    cipher: 'chacha20-poly1305',
    key: sharedSecret,
    natTraversal: true // ICE/STUN/TURN as needed
  })

  // 5. Exchange full-fidelity embeddings to confirm match
  const trueSimilarity = cosineSim(myTrueEmbedding, channel.receive())
  if (trueSimilarity < CONFIRM_THRESHOLD) {
    channel.close('match_not_confirmed')
    return null
  }

  return channel // ready for structured exchange
}
NAT Traversal

Most personal nodes sit behind NATs and firewalls. Direct P2P connections use ICE (Interactive Connectivity Establishment) with STUN for discovery and TURN as fallback relay. The TURN relay is only a transport proxy — it forwards encrypted bytes it cannot read. Community-operated TURN servers are part of the infrastructure contribution model.

Chapter 7

Routing Protocol: Finding Your Shard

Routing in Resonance answers a simple question: given a perturbed embedding, which relay nodes should receive it? And given a live search, which shards should be queried? This is the plumbing that makes network-wide discovery work without centralized coordination.

Network Layer: libp2p

Resonance uses libp2p as its networking layer — the same protocol stack that powers IPFS, Filecoin, and Ethereum 2.0. libp2p provides peer discovery, NAT traversal, multiplexed connections, and a Kademlia-based DHT for routing.

The Routing Table

Every personal node maintains a local routing table that maps shard IDs to current relay nodes. This table is updated every epoch when relays rotate:

// Routing table structure
type RoutingTable = {
  epoch: number,
  shards: Map<ShardId, {
    relays: PeerId[], // ordered by latency
    vectorSpace: string, // which embedding family
    lshParams: LSHConfig // for shard assignment
  }>,
  ttl: number // refresh before this timestamp
}

// Routing table is published to DHT by relay coordinators
// Personal nodes subscribe to routing updates via pubsub
const ROUTING_TOPIC = '/planetary/resonance/routing/v1'

Publish Routing

When you publish an embedding, the routing path is:

R1

→
R2

Check local routing table for shard's relay set
→
R3
Multicast

→
R4

Search Routing

Live searches follow a similar path but with two modes:

Mode Shards Queried Latency Recall Privacy Cost
Targeted Primary shard + 2 neighbors ~100ms ~85% Low (1 query seen by 3 relays)
Broadcast All shards ~500ms ~98% Higher (query seen by all relays)

The default is targeted search. Broadcast is available as a premium feature (costs network credits) for when completeness matters more than speed or privacy.

Cross-Region Routing

As the network scales to multiple regions, shards become geo-aware. Each shard is tagged with a region affinity (but not restricted to it). A housing search in Barcelona routes preferentially to shards with Mediterranean-region embeddings, but can fall back to global shards if local results are insufficient.

Why Not Pure DHT?

A pure Kademlia DHT would route by content hash, not by vector similarity. Two semantically similar embeddings would hash to completely different parts of the DHT. Locality-sensitive hashing (LSH) bridges this gap: similar vectors hash to the same bucket with high probability, so the shard assignment preserves the semantic proximity that makes ANN search work.

Progressive Scaling

Resonance is designed to start small and scale without rewrites. The architecture is the same at 100 nodes and 10 million — only the parameters change.

What Changes vs. What Doesn't

Never Changes

Raw data stays local. Embeddings are perturbed. Relays are ephemeral. Direct channels are E2E encrypted. The three-tier model holds at any scale.

Scales Horizontally

Number of shards. Number of relays per shard. Relay rotation frequency. HNSW parameters. Replication factor. All configurable, all additive.

Emerges at Scale

Hierarchical shard trees. Cross-region super-relays. Multi-model projection layers. Tiered search (free vs. credit-based). Governance-driven parameter tuning.

Chapter 9

Security Model: What Can Go Wrong

Every distributed system has attack surfaces. Resonance is no exception. The architecture is designed to make attacks expensive, detectable, and limited in blast radius. Here's what we defend against and how.

Attack

Malicious Relay

Threat: A relay logs all embeddings, analyzes patterns, or selectively drops matches.
Defense: Relay rotation limits exposure to 24-72h. Sharding means each relay sees <7% of traffic. Replication means dropped matches are caught by other replicas. Users can verify match completeness via cross-shard queries.

Attack

Embedding Inversion

Threat: An attacker tries to reconstruct raw text from a perturbed embedding.
Defense: Two layers: (1) embedding models are lossy by design — many texts map to similar vectors; (2) differential privacy noise makes inversion exponentially harder. At ε=1.0, successful reconstruction probability is <0.1%.

Attack

Sybil Attack

Threat: An attacker creates thousands of fake nodes to dominate relay selection or flood the index.
Defense: Relay selection requires minimum uptime + stake (compute contribution history). New nodes have a probationary period. VRF-based selection is resistant to manipulation.

Attack

Correlation Attack

Threat: An observer monitors relay traffic to correlate who publishes when, linking DIDs to IP addresses.
Defense: Nodes publish via mixnet-style batching — embeddings are queued and published in shuffled batches every 5-15 minutes. Optional Tor/I2P integration for transport-level anonymity.

Attack

Index Poisoning

Threat: An attacker floods the index with garbage embeddings to degrade search quality.
Defense: Rate limiting per DID. Reputation-weighted indexing (new DIDs get lower index priority). Anomaly detection on embedding distributions. Community reporting and blacklisting via governance.

Attack

Relay Collusion

Threat: Multiple relays collude to reconstruct a larger picture of the network.
Defense: VRF relay selection is random — you can't choose your co-relays. Shard assignments rotate independently. Cross-shard data has different perturbation seeds. Colluding relays would need to control a majority of shards simultaneously, which becomes harder as the network grows.

Honest Limitation

No system is perfectly private. A sufficiently motivated state-level adversary with persistent control of a large fraction of relay nodes could learn aggregate patterns about the network (e.g., "housing queries are increasing in Barcelona"). The architecture makes this expensive and limits the granularity of what can be learned, but it doesn't make it impossible. We are honest about this. The goal is to make mass surveillance impractical and individual targeting extremely difficult — not to promise mathematical impossibility against nation-state adversaries.
