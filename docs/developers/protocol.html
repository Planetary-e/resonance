Wire Protocol v0.1

Wire Protocol

The complete specification for messages exchanged between personal nodes, relays, and direct channels. All messages are signed with ed25519 and transported over TLS WebSockets.

Envelope

Message Envelope

Every message in the Resonance protocol uses a common envelope format. The payload is generic and varies by message type.

interface Message<T> { type: string; // Message type identifier from: string; // DID of sender (did:key:z...) timestamp: number; // Unix milliseconds signature: string; // Base64-encoded ed25519 signature payload: T; // Type-specific payload }
Signing

The signature field is computed over JSON.stringify({ type, from, timestamp, payload }) using the sender's ed25519 secret key. Recipients verify using the public key derived from the sender's DID.

Node → Relay

Node to Relay Messages

Messages sent from a personal node to a relay for publishing, searching, consenting, and withdrawing.

publish

Publish a perturbed vector to the relay's index.

interface PublishPayload { itemId: string; // Unique identifier for this item vector: number[]; // 768-dimensional perturbed vector itemType: "need" | "offer"; // Whether this is a need or an offer ttl: number; // Time-to-live in seconds (max 86400) }

search

Search the relay's index for matching items. The query vector is a true (un-noised) embedding.

interface SearchPayload { vector: number[]; // 768-dimensional true embedding (query) k: number; // Maximum number of results to return threshold: number; // Minimum cosine similarity (default: 0.50) }

consent

Accept or decline a match. Optionally includes encrypted data for the partner.

interface ConsentPayload { matchId: string; // Match ID from the match notification accept: boolean; // true to accept, false to decline encryptedForPartner: string?; // Optional: encrypted ephemeral public key }

withdraw

Remove a previously published item from the relay's index.

interface WithdrawPayload { itemId: string; // ID of the item to remove }
Relay → Node

Relay to Node Messages

Messages sent from the relay back to personal nodes: match notifications, search results, consent forwarding, and acknowledgments.

match

Notify a node that one of its published items matched another user's item.

interface MatchPayload { matchId: string; // Unique match identifier partnerDID: string; // DID of the matched partner similarity: number; // Cosine similarity (on perturbed vectors) yourItemId: string; // Your item that matched partnerItemType: "need" | "offer"; // Whether the partner's item is a need or offer expiry: number; // Unix ms when this match expires }

search_results

Return results from a search query.

interface SearchResultsPayload { results: { did: string; // DID of the matching item's publisher similarity: number; // Cosine similarity score itemType: "need" | "offer"; // Type of the matching item }[]; }

consent_forward

Forward a partner's consent decision (and optional encrypted payload).

interface ConsentForwardPayload { matchId: string; // Match ID fromDID: string; // DID of the partner who consented encrypted: string?; // Optional encrypted data from the partner }

ack

Acknowledge receipt of a message.

interface AckPayload { ref: string; // ID of the message being acknowledged status: "ok" | "error"; // Whether the operation succeeded message?: string; // Optional human-readable error detail }
Direct Channel

Direct Channel Messages

After both parties consent and establish a shared secret via X25519 key exchange, they communicate directly. These messages are end-to-end encrypted using the derived symmetric key.

confirm_embedding

Send the true (un-noised) embedding vector for the matched item.

interface ConfirmEmbeddingPayload { vector: number[]; // 768-dimensional true embedding }

confirm_result

The result of comparing true embeddings from both sides.

interface ConfirmResultPayload { similarity: number; // Cosine similarity of true embeddings confirmed: boolean; // true if similarity >= 0.55 }

disclosure

Progressively disclose information, from category to detail to contact info.

interface DisclosurePayload { text: string; // Disclosed content level: "category" | "detail" | "contact"; // Disclosure level }
Progressive Disclosure

Disclosure happens in stages. First, share a general category ("housing in Barcelona"). If the other party is still interested, share details ("2-bedroom flat in Gracia, available April"). Finally, share contact info if both parties want to proceed. Each step is a conscious choice by the user.

accept

Accept the exchange and optionally include a message.

interface AcceptPayload { message?: string; // Optional message to the partner }

reject

Decline the exchange.

interface RejectPayload { reason?: string; // Optional reason (not required) }

close

Close the direct channel. Either party can close at any time.

interface ClosePayload { // Empty — closing the channel requires no additional data }
