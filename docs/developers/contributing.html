Contributing

Resonance is open source and welcomes contributions. Whether you are fixing a bug, adding a benchmark, or improving documentation, here is everything you need to get started.

Prerequisites

What You Need

Node.js 20+

The project uses modern ESM features and requires Node.js 20 or later. Check your version with node --version.

npm Workspaces

The monorepo uses npm workspaces to link packages together. All dependencies are installed from the root.

Git

Standard git workflow. Fork the repository, create a feature branch, submit a pull request.

Setup

Development Setup

Get the project running locally in a few steps.

# Clone the repository git clone https://github.com/Planetary-e/resonance.git cd resonance # Install all dependencies (monorepo-wide) npm install # Verify everything works npx vitest run
First Run

The first time you run tests or eval, the embedding model (~90MB) will be downloaded and cached. Subsequent runs start in under a second.

Structure

Project Structure

The monorepo is organized into four packages, each with a focused responsibility.

resonance/ packages/ core/ // @resonance/core — The heart of the system src/ types.ts // Shared types, constants, config crypto.ts // Identity, signing, encryption perturbation.ts // Laplace noise / differential privacy embedding.ts // HuggingFace transformer embeddings vectors.ts // Vector math (cosine sim, normalize) protocol.ts // Wire protocol messages relay/ // @resonance/relay — Relay server src/ matching-index.ts // Separate HNSW for needs/offers relay-server.ts // WebSocket server + admin API node/ // @resonance/node — Personal node CLI src/ cli.ts // Command-line interface store.ts // Local encrypted key-value store relay-client.ts // WebSocket client for relay channel.ts // Direct channel management eval/ // @resonance/eval — Evaluation and benchmarks src/ benchmarks/ // Individual benchmark modules run.ts // Main runner + report generator docs/ // Documentation site (this site) assets/style.css // Shared stylesheet evals/ // Auto-generated eval reports scripts/ dev-cluster.sh // Local development cluster launcher
Testing

Running Tests

Unit Tests (119 tests)

The project has 119 unit tests covering crypto, perturbation, vectors, protocol, store, relay, and channels.

# Run all tests npx vitest run # Run tests in watch mode npx vitest # Run tests for a specific package npx vitest run --filter core npx vitest run --filter relay npx vitest run --filter node

Evaluation Suite (35 metrics)

The eval suite is separate from unit tests. It runs real benchmarks with real data and generates reports.

# Quick eval: 10K vectors, ~2 minutes npm run eval:quick # Full eval: 100K vectors, ~15 minutes npm run eval # Reports are saved to docs/evals/eval-YYYY-MM-DD-HH-MM-SS.md

Local Demo

Development Cluster

Run a local multi-node cluster to test the full flow end-to-end: relay server, two personal nodes, matching, consent, and channel exchange.

# Launch a local dev cluster (relay + 2 nodes) bash scripts/dev-cluster.sh

The dev cluster starts a relay on ws://localhost:8080, creates two personal nodes with ephemeral identities, publishes a need and an offer, and walks through the full matching flow including consent, confirmation, disclosure, and channel close.

What It Does

The script launches a relay server, two node instances, publishes sample items, triggers matching, and demonstrates the complete consent → confirm → disclose → accept flow. All output is printed to the terminal with timing information.

Guidelines

Code Style

Keep these guidelines in mind when contributing.

TypeScript Strict

All code must pass tsc --strict. No any types, no implicit returns, no unused variables. The compiler is your friend.

ESM Only

Pure ECMAScript Modules throughout. Use .js extensions in imports (even for TypeScript files). No require().

No GPL Dependencies

All dependencies must be MIT, Apache-2.0, or BSD licensed. No GPL or LGPL. Check before adding a new dependency.

Test Everything

New features need unit tests. New algorithms need eval benchmarks. If it affects matching quality, it should show up in the eval report.

Important

Never commit real embedding vectors, private keys, or personal data to the repository. Test data should be synthetic or clearly marked as sample data. Eval benchmarks use curated test pairs, not real user data.

Benchmarks

Adding a Benchmark

To add a new benchmark to the eval suite, follow this pattern.

1. Create the Benchmark File

Create a new file in packages/eval/src/benchmarks/:

// packages/eval/src/benchmarks/my-benchmark.ts import type { BenchmarkResult } from '../types.js'; export async function runMyBenchmark(): Promise<BenchmarkResult[]> { const results: BenchmarkResult[] = []; // Your benchmark logic here... results.push({ metric: 'My metric name', target: '>0.50', // or 'info' for tracking-only actual: '0.72', pass: true, // or null for info metrics }); return results; }

2. Wire into run.ts

Import and call your benchmark in the main runner:

// packages/eval/src/run.ts import { runMyBenchmark } from './benchmarks/my-benchmark.js'; // Add to the benchmark list const benchmarks = [ // ...existing benchmarks... { name: 'My Benchmark', fn: runMyBenchmark }, ];

3. Run and Verify

npm run eval:quick # Check the generated report in docs/evals/

Pull Request Tips

Include the output of npm run eval:quick in your PR description when your change affects matching quality, privacy, or performance. This makes review much faster. All 35 metrics must pass before merge.
