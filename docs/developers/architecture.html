Technical Reference

Architecture

A three-tier system where personal nodes handle computation and privacy, relays handle anonymous discovery, and direct channels handle verified exchange. Every design decision is validated by evaluation data.

Overview

Three-Tier Architecture

Resonance separates concerns across three trust boundaries. Data visibility decreases as you move from the personal node outward.

Tier 1
Personal Node
User's device
Full data visibility
Embedding + perturbation
↓ noisy vectors + DIDs ↓
Tier 2
Relay
Community server
Perturbed vectors only
HNSW index + matching
↓ match notification ↓
Tier 3
Direct Channel
Peer-to-peer encrypted
True embeddings + disclosures
User-controlled

Design Decisions

Key Design Decisions

Every major design choice in Resonance was validated by the evaluation suite. Here are the decisions that most impact matching quality and privacy.

Index-Only Perturbation

Only the indexed vectors (published to the relay) receive Laplace noise. When a personal node searches the relay, it uses the true embedding as the query vector. This asymmetric approach preserves search quality while keeping indexed data private.

Eval Evidence

Index-only perturbation at epsilon=1.0 achieves 82–86% recall at threshold 0.50. Double perturbation (both index and query noised) at the same epsilon drops to 48% recall and requires lowering the threshold to 0.30 for full recall—which produces unacceptable noise.

Relay Threshold: 0.50

The relay's cosine similarity threshold was lowered from the original 0.72 to 0.50 based on evaluation data. This intentionally over-matches at the relay level, relying on the confirmation step to filter false positives.

Eval Evidence

At threshold 0.72, recall was only 4% (2/50 pairs matched). At 0.50, recall jumps to 94% for baseline and 82–86% for the MatchingIndex with perturbation. The confirmation step at 0.55 on true embeddings filters the false positives.

Confirmation Threshold: 0.55

After both parties consent, they exchange true (un-noised) embeddings over the encrypted direct channel and verify that the cosine similarity exceeds 0.55. This catches false positives from the noisy relay matching.

Eval Evidence

The two-stage design (relay at 0.50, confirm at 0.55) achieves <3% FPR with the MatchingIndex while maintaining 82–86% recall. The confirmation step adds only 25ms latency.

MatchingIndex: Separate Need/Offer Indexes

The relay maintains two separate HNSW indexes: one for needs and one for offers. Needs are only searched against offers and vice versa.

Eval Evidence

A flat (single) index produced 131 matches at epsilon=1.0 and threshold 0.50, but 65% of those matches were same-type noise (needs matching needs, offers matching offers). The MatchingIndex eliminated this entirely, reducing matches to 47 with only 2.1% false positive rate.

Query Rewriting

Need-type items are rewritten to strip demand framing before embedding. "I'm looking for a Django developer" becomes "Django developer" so it matches offer-side descriptions more accurately.

Eval Evidence

Rewriting improved average need-offer similarity from 0.634 to 0.650 (+1.5pp) and boosted recall from 80% to 86% at epsilon=1.0 with threshold 0.50.

Relay-Bridged Channels

Direct channels are bridged through the relay rather than requiring NAT traversal. Messages are E2E encrypted—the relay forwards opaque ciphertext without being able to read content. This simplifies deployment for the pilot while preserving privacy guarantees.

Eval Evidence

Relay-bridged channels achieve 9.5ms message round-trip, 16ms consent handshake, and 25ms confirmation exchange. The relay sees only encrypted blobs and routing DIDs—never plaintext.
